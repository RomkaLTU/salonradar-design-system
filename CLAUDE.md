# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Two things in one **npm-workspaces** monorepo:

1. **`packages/salonradar-ui`** (`@salonradar/ui`) — the publishable design system, **published to npmjs**. A warm coral + cream theme built on shadcn/ui (new-york style), Radix UI, and Tailwind CSS v4. ~55 accessible React components, OKLCH design tokens, light/dark themes, a `ThemeProvider`, and a `Logo`.
2. **Root Next.js app** (`salonradar-starter`) — a showcase/docs site that consumes the **published-shape** package (its built `dist/`) and renders the tokens + component gallery (`app/`, `components/`). It doubles as the "real consumer app" that [v0 design systems](https://v0.app/docs/design-systems-2) expect as a reference.

The repo is linked to a [v0](https://v0.app) project; v0 may push commits directly, and merges auto-deploy on Vercel.

## Package manager: npm (not pnpm)

This repo uses **npm with workspaces** (`workspaces: ["packages/salonradar-ui"]` in the root `package.json`, `package-lock.json` lockfile). Do **not** reintroduce pnpm — it was removed because pnpm's `publishConfig` field-override behavior (see below) and supply-chain install restrictions were breaking install + publish. There must be no `pnpm-lock.yaml` / `pnpm-workspace.yaml`.

npm workspaces symlinks `node_modules/@salonradar/ui` → the real `packages/salonradar-ui` directory (the root dep is `"@salonradar/ui": "*"`), so its `dist/` is resolved live.

## Commands

```bash
npm install           # installs root app + workspace package, links the workspace
npm run build:ui      # build ONLY the design system → packages/salonradar-ui/dist (tsc)
npm run dev           # predev builds the UI package, then next dev (http://localhost:3000)
npm run build         # build:ui, then next build  (this is what Vercel runs)
npm start             # serve the production build
npm run lint          # next lint
```

Iterating on the design system with live rebuilds:

```bash
npm run dev --workspace @salonradar/ui   # tsc --watch → keeps dist/ fresh while editing src
```

There is no test framework configured.

## Build & consumption model (important — changed from the original setup)

The app consumes the design system's **compiled `dist/`**, exactly like an external npm consumer would:

- `@salonradar/ui` `exports` / `main` / `types` point to `./dist/*` (`.js` + `.d.ts`). The app imports compiled JS, so `next.config.mjs` has **no** `transpilePackages` and needs none.
- `dist/` must exist before the app builds. The root `build` and `predev` scripts run `build:ui` first, so `npm run build` / `npm run dev` are self-contained. Editing a component in `src/` does **not** hot-reload into the app unless you also run the package's `tsc --watch` (see above).
- `dist/` is git-ignored build output — never edit it by hand; it is regenerated by `build:ui` (`rm -rf dist && tsc -p tsconfig.build.json`).
- Vercel build is pinned in `vercel.json` (`installCommand: npm install`, `buildCommand: npm run build`) so the package is always built before the Next app.

### Publishing to npm (the field-override gotcha)

The package is published with plain `npm publish` (from `packages/salonradar-ui`, or `npm publish -w @salonradar/ui`). `prepublishOnly` rebuilds `dist/`.

**Do not move `main`/`types`/`exports` into `publishConfig`.** pnpm applies `publishConfig` field-overrides at publish time; **npm does not** — it ignores them (warns "Unknown publishConfig config"). The original package relied on `publishConfig.exports` → `dist` and shipped only `dist` in `files`, so under npm the published `package.json` still pointed `exports` at unshipped `./src/*` and was completely broken. The real top-level `exports` now point to `dist`; `publishConfig` holds only `{ "access": "public" }`. Verify any change with `npm pack` + inspect `package/package.json`, confirming every `exports` target exists in `npm pack --dry-run`'s file list.

When adding a new UI component: create it under `src/components/ui/`, add its `export *` line to `src/index.ts`, then `npm run build:ui`. The `./*` export pattern maps `@salonradar/ui/<name>` → `dist/components/ui/<name>.js` automatically.

## Theme & Tailwind v4 wiring

- The theme source of truth is `packages/salonradar-ui/styles/globals.css`: OKLCH tokens for `:root` and `.dark`, `@custom-variant dark`, and the `@theme inline` block mapping tokens to Tailwind color/font utilities and the radius. It ships as-is (`files` includes `styles`) and is imported via `@salonradar/ui/styles.css`.
- The app's `app/globals.css` is order-sensitive: `@import 'tailwindcss'` → `@import 'tw-animate-css'` → `@import '@salonradar/ui/styles.css'`, then **`@source '../packages/salonradar-ui/src'`**. That `@source` line is required so Tailwind v4 scans the design system for class names; without it those classes get purged. (Class strings are identical in `src` and `dist`, so scanning `src` is correct and always present.) External consumers must add their own `@source` pointing at the installed package's `dist`.
- Fonts are loaded by the **host app** (here `next/font` in `app/layout.tsx`: Geist → `--font-geist`, Instrument Serif → `--font-instrument-serif`), mapped by the theme to `--font-sans` / `--font-serif`. The package bundles no fonts.
- Light/dark is driven by `next-themes` via the package's `ThemeProvider` (re-exported from `@salonradar/ui/theme-provider`).

## Conventions

- Components are shadcn/ui (new-york) style: CVA variants, Radix primitives, `cn()` (from `@salonradar/ui/lib/utils` — `clsx` + `tailwind-merge`) for class composition. Match this when adding or editing components.
- Inside the package, imports are **relative** (`../../lib/utils`) — no `@/` alias (it would not resolve in `dist`). The root app uses the `@/*` alias (→ repo root).
- Style with token-backed utilities (`bg-background`, `text-foreground`, `text-primary`, `border-border`, `font-serif`, …) rather than hard-coded colors, so theming and light/dark keep working.
